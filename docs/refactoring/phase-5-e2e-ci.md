# Phase 5: E2E テストと CI 品質ゲート

**ゴール**: リファクタリング成果を回帰から守る自動化を整備し、品質ゲート（カバレッジ・E2E）を実効性のある水準に引き上げる。

**前提**: Phase 1 完了（テストが実コードを検証している）。Phase 2〜4 完了後の実施が理想だが、E2E の整備自体は Phase 2〜4 と並行可能。

## 背景（2026-06-11 調査）

- `e2e/` の 2 スペックは全テストが `test.skip(!process.env.E2E_TEST_ENABLED, ...)` でスキップされるスケルトン。CI にも組み込まれていない。`playwright.config.ts` の `baseURL`（8080）も実サーバーと不一致（Phase 0 で修正済みの想定）。
- `jest.config.ts` のカバレッジ閾値はフロント statements 3% / branches 1% / functions 0%。フロントは `collectCoverageFrom: []` で計測対象すら空（Phase 1 で計測開始済みの想定）。
- `.github/workflows/test.yml` の backend-tests は `|| echo` で失敗を握りつぶしていた（Phase 0 で修正済みの想定）。codecov の目標値も 1% と形骸化。
- E2E がないため、フロントの DOM 操作系（モーダル・タブ切り替え・申請フロー）はリリース前の手動確認に依存している。

## 作業項目

### 5-1. E2E の土台整備（PR 1 本）

- [ ] テスト用のシード戦略を決める: テスト起動時に一時 SQLite（専用 `DATABASE_PATH`）を作成し、シードデータを投入するグローバルセットアップを実装
- [ ] 認証: `devLogin.js`（開発用ログイン）を E2E から利用できるようにする。本番ビルドに影響しないことを確認
- [ ] Google Calendar 連携はテスト時に無効化（環境変数未設定なら同期スキップ、または getCalendarClient のスタブ化）し、外部 API に依存しない構成にする
- [ ] `test.skip` ガードを撤去し、ローカルで `npx playwright test` が素で通る状態にする

### 5-2. 主要フローの E2E 実装（PR 2〜3 本）

優先度順:

- [ ] **シフト申請フロー**: ログイン → 日付選択 → 時間枠選択 → 申請 → 自分のシフトに表示される
- [ ] **申請キャンセル**: 申請済みシフトの削除 → 一覧から消える → 残り枠が戻る
- [ ] **特別シフト**: 管理者が特別シフト作成 → 一般ユーザーが 30 分スロット申請 → 一覧表示
- [ ] **人数設定（管理者）**: 容量変更 → 残り枠表示への即時反映
- [ ] Phase 1 で「E2E でカバーする」と判断して削除した DOM 系テストの対象がすべて含まれているか、`test-inventory.md` と突き合わせて確認

### 5-3. CI への組み込み（PR 1 本）

- [ ] `.github/workflows/test.yml` に E2E ジョブを追加（Playwright 公式 action、失敗時に trace/screenshot をアーティファクト保存）
- [ ] E2E は `pull_request` 時に必須チェック化（実行時間が 5 分を超えるようなら main マージ時のみに調整）

### 5-4. カバレッジ閾値の実効化（PR 1 本）

- [ ] 現状のカバレッジ実測値を確認し、「実測値 − 数 %」を新しい閾値として設定（落ちたら即 fail する、形骸化しない値にする）
  - 目安: バックエンドは現状 95% 設定を維持、フロントは Phase 1〜3 完了後の実測に基づき 60% 以上を目標
- [ ] `codecov.yml` の target 1% を実測ベースに引き上げ、`patch` カバレッジ（新規変更行）も設定
- [ ] 以後、閾値はフェーズ完了ごとに実測へ追随させる運用を CLAUDE.md に明記

### 5-5. 仕上げ（PR 1 本）

- [ ] `docs/TESTING.md` を更新: テスト 3 層（unit / integration / E2E）の役割分担、ローカル実行方法、シード戦略
- [ ] この `docs/refactoring/` 全体の完了状況を README.md の進捗表に反映し、残課題（やらないと決めたことを含む）を `docs/refactoring/leftovers.md` に記録

## リスクと対策

| リスク | 対策 |
|--------|------|
| E2E がフレーキーで CI を阻害する | リトライ 1 回 + trace 保存を設定。フレーキーなテストは即 quarantine（skip + issue 化）するルールを決めておく |
| dev ログインを E2E に使うことで本番に穴が開く | devLogin の有効化条件（NODE_ENV / 環境変数）を確認し、E2E 用フラグが本番で立たないことをテストで保証 |
| カバレッジ閾値を上げすぎて開発が止まる | 「実測 − 数 %」ルールを厳守。理想値ではなく現実の床として運用 |

## 完了条件（Definition of Done）

- 主要 4 フローの E2E が CI で毎 PR 実行され、green
- カバレッジ閾値が実測ベースで、下回ると CI が fail する
- `docs/TESTING.md` が 3 層構成を反映している
- リファクタリング全体の完了宣言と残課題リストが残っている
